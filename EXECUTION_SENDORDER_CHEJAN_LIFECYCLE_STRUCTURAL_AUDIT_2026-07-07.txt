Execution / SendOrder / Chejan / Lifecycle Pipeline Structural Audit
Date: 2026-07-07
Scope: structural audit only. No Python code changes, no runtime writes, no rules changes, no layer deletion or movement.

1. Audit Target Summary

This audit reviews the currently accumulated execution pipeline layers:

- Execution Preview and Readiness layers
- Queue Commit preview, dry-run, executor, and review layers
- SendOrder dispatch, adapter, safety, executor, and review layers
- SendOrder result recorder contract, recorder, and review layers
- Chejan entry, raw event, classification, evidence, and evidence review layers
- Chejan-to-Lifecycle transition preview, approval, commit contract preview, and dry-run layers

The core pattern is consistent:

Preview -> Readiness/Approval -> Contract -> Dry-run -> Executor -> Review -> Next boundary

This pattern is safe for automatic trading, but the number of layers is now high enough that naming, grouping, and commit boundaries should be managed deliberately.

3. Keep, But Consider Name or Role Adjustment

- rule_apply_preview_execution_order_adapter.py
  Current role is useful, but name is long. Consider grouping under a "rule_to_execution" naming family later.

- rule_apply_preview_execution_preview_controller.py
  Useful as a controller-level preview hook. Consider naming it as an orchestrator if it remains non-GUI and preview-only.

- execution_queue_review_to_send_order_preview_adapter.py
  Useful adapter. Name is accurate but very long; consider grouping under "send_order_preview_adapters" if modules are later packaged.

- execution_final_send_gate_* layers
  They provide a safe bridge to final_send_gate_service. If the final gate remains a single function, some input adapter/orchestrator policy layers may be grouped later.

- execution_send_order_entrypoint_* layers
  Keep while SendOrder entrypoint remains a distinct pre-broker boundary. Revisit once actual broker dispatch path stabilizes.

- execution_broker_dispatch_open_policy.py and execution_broker_dispatch_orchestrator.py
  Keep, but ensure they do not duplicate kiwoom_send_order_executor responsibilities. Broker dispatch and Kiwoom SendOrder executor need a clear separation:
  broker dispatch = injected broker adapter call
  Kiwoom SendOrder executor = direct SendOrder-like injected callable call

- execution_runtime_status_update_* and execution_queue_status_update_* layers
  Keep as preview-only post-broker orchestration, but do not confuse them with real runtime/order_queue file updates.

- execution_lock_release_* and execution_post_execution_review.py
  Keep for complete preview chain testing. Real lock release must remain separate from preview release.

4. Integration Candidates

These are candidates for future consolidation after behavior stabilizes. Do not merge now without tests and spec approval.

- Final Send Gate readiness + input adapter + preview orchestrator
  Potential consolidation: readiness policy and input adapter could become one "final_send_gate_input_preview" layer if responsibilities remain simple.

- Final Send Gate open policy + call orchestrator
  Potential consolidation: could become one "final_send_gate_service_preview_orchestrator" only if policy remains small and no environment flags expand.

- SendOrder EntryPoint open policy + SendOrder EntryPoint orchestrator
  Potential consolidation: if entrypoint remains preview-only and never calls broker, policy and orchestrator may be combined.

- Broker Result Review + Broker Result Record Readiness Policy
  Potential consolidation: review status could directly produce record readiness if no independent operator/environment policy is required.

- Runtime Status Update Readiness + Runtime Status Update Orchestrator
  Potential consolidation: if updater stays in-memory preview-only, readiness may be a section of the orchestrator.

- Queue Status Update Readiness + Queue Status Update Orchestrator
  Same as runtime status update.

- Lock Release Readiness + Lock Release Orchestrator
  Same pattern; safe but possibly verbose.

- Chejan Event Classification Preview + Evidence Builder
  Keep separate now because classification and evidence have different meanings. If evidence remains a thin wrapper, they can later become one "classification_evidence_preview" layer.

- Lifecycle Approval Gate + Lifecycle Commit Contract Preview
  Keep separate now because approval and contract-building are distinct. Consolidation should wait until a real lifecycle writer exists.

10. Dead Layer and Over-Layering Risk

Risk exists in the post-queue and pre-SendOrder region:

- Final Send Gate readiness/input/orchestrator/open/call chain is very granular.
- SendOrder EntryPoint open/orchestrator chain may overlap with Kiwoom safety gate and call preview.
- Broker Dispatch open/orchestrator may overlap with Kiwoom SendOrder executor if both remain active paths.
- Runtime status update and queue status update preview layers may be conceptual until real writers exist.

Practical interpretation:

- Keep the layers for now because they are covered by tests and preserve explicit boundaries.
- Avoid adding new adjacent policy/orchestrator pairs unless they introduce a genuinely new safety boundary.
- Once the real Kiwoom path is chosen, collapse redundant pre-call layers only with contract tests in place.

11. Stability Contribution

Strong stability contributors:

- Readiness validators
- Approval gates
- Contract previews
- Dry-runs
- Executor boundaries
- Result reviews
- Explicit injected adapters
- Hash/immutability tests

Lower stability contributors if left too granular:

- Multiple open policy layers that only restate the previous result
- Orchestrators that only wrap one function without a new safety check
- Preview layers that duplicate the same identity checks without adding new context

12. Kiwoom Integration Practicality

The current path is safe but verbose. For real Kiwoom integration:

- Keep one final safety gate immediately before SendOrder.
- Keep one call preview that materializes exact SendOrder args.
- Keep one executor that calls only an injected callable.
- Keep one result review that classifies return values.
- Keep recorder and Chejan pipeline strictly separated.

Natural real integration path:

Queue Commit Review
-> Dispatch Builder
-> Broker Dispatch Preview
-> Kiwoom Adapter Contract
-> Kiwoom Safety Gate
-> Kiwoom Call Preview
-> Kiwoom SendOrder Executor
-> Executor Result Review
-> Result Recorder Contract
-> Result Recorder
-> Recorder Review
-> Chejan Entry Adapter

The older Final Send Gate and EntryPoint chain should be evaluated for redundancy against this newer direct Kiwoom contract path.

13. Host / Local Extension

The layered contract style is favorable for Host/Local operation:

- Local Ollama/OpenCode can analyze preview contracts without write access.
- Host-controlled Codex can implement gated write layers.
- Injected broker adapters allow mock/local tests and production adapters to remain separate.
- Report-only and preview-only stages are easy to run safely on local machines.

The cost is naming and navigation complexity. A future package layout may help, but no restructuring should happen until the active pipeline is stable.

16. Practical Judgment

The pipeline is safe-first and test-heavy, which is appropriate for automatic trading. The main issue is not unsafe behavior but layer sprawl.

Current best action:

- Freeze new conceptual expansion.
- Pick one canonical SendOrder path.
- Add contract tests that prove the canonical path reaches the intended boundary.
- Only then decide which redundant pre-SendOrder layers can be merged or retired.
